# 업무보고 일지 작성 (SPM)

$ARGUMENTS 날짜의 업무보고를 작성한다. 인자 없으면 오늘 날짜를 사용한다.

특정 프로젝트 전용 경로를 가정하지 않는다. 항상 현재 작업 디렉토리의 저장소/프로젝트를 기준으로 작성한다.

## 절차

1. 현재 위치 확인: `pwd`, 필요하면 `git rev-parse --show-toplevel`
2. 작업 내용 확인: `git status --short`, `git log --oneline -10`, 필요하면 최근 diff/stat 확인
3. 보고서 저장 위치 결정
4. `업무보고_YYYYMMDD.md` 파일 작성
5. 끝. 작성 파일 경로를 보여주고 멘트 한 줄.

## 저장 위치 결정

아래 순서로 가장 먼저 맞는 위치를 사용한다.

1. 기존 `docs/reports/업무보고/` 디렉토리가 있으면 사용
2. 기존 `docs/reports/daily/` 디렉토리가 있으면 사용
3. 기존 `docs/daily/` 디렉토리가 있으면 사용
4. 기존 `reports/` 디렉토리가 있으면 `reports/업무보고/` 생성 후 사용
5. 아무것도 없으면 `docs/reports/업무보고/` 생성 후 사용

파일명은 항상 `업무보고_YYYYMMDD.md`를 사용한다.

## 형식

```
# 업무보고 - YYYY-MM-DD

## 업무내용
1) 큰 분류 1
2) 큰 분류 2

## 진행상황

### 1.1) 작업명
- 핵심 한 줄
- 결과/수치 (선택)

### 2.1) 작업명
- 핵심 한 줄

## To do list
1) 다음 작업
2) 다음 작업
```

## 작성 원칙 — 짧고 명료하게

### 길이 제한

- 작업 항목당 **bullet 최대 2개**
- bullet 1개당 **한 줄**
- 전체 보고서 **30줄 이내** (formatting 제외)

### 군더더기 금지

- "~를 진행했음", "~를 작업했음" 같은 동사 늘이기 X → "~완료", "~정상 동작" 으로 끊기
- "~할 예정", "~할 계획" 같은 미래형 잡담 X — To do list에만 넣기
- 배경/이유 설명 X — 팀장님은 결과만 본다
- 코드 스니펫, 명령어, 파일 경로 줄줄이 X — 핵심만 한 줄로

### 좋은 예 / 나쁜 예

❌ 나쁜 예 (장황):

```
### 1.1) 카메라 hot-reload 기능 작업
- 기존에 카메라를 추가할 때 docker compose restart를 해야 해서 5초 다운타임이 있었는데, 이를 개선하기 위해 go2rtc fork에 새로운 API endpoint를 추가하는 작업을 진행했음
- internal/streams/api.go 파일에 apiStreamsReload handler를 만들고, internal/streams/streams.go에 reconcileStreams 헬퍼를 작성해서 메모리와 yml을 비교한 후 diff를 적용하는 로직을 구현함
- 추가로 보안을 위해 localhost-only 검증을 넣었고, scripts/rtc-apply.sh 스크립트도 작성함
```

✅ 좋은 예 (간결):

```
### 1.1) rtc fork에 reload API 추가
- `POST /api/streams/apply` 엔드포인트 추가 — 컨테이너 재시작 없이 yml 반영
- localhost-only 보안 + `scripts/rtc-apply.sh` 운영 스크립트
```

### 결과 우선 표현

- "~했음" → "~완료"
- "구현 중" → To do list로
- "~ 시도" → 결과까지 적기 ("성공" 또는 "실패: 이유")
- 수치 있으면 무조건 넣기: "5초 → 0초", "17줄 → 9줄"

### 번호 매칭

- 업무내용 `1)`, `2)` → 진행상황 `1.1`, `1.2`, `2.1` 식으로 매칭

### 용어

- 팀장님이 이해할 수 있는 수준. 약어는 한 번만 풀어 쓰기
- 코드 변수명/함수명 줄줄이 나열 X — 결과만

## 프로젝트 독립성

- 특정 회사명, 프로젝트명, 절대경로를 스킬 본문에서 가정하지 않는다.
- 저장소가 아니어도 현재 디렉토리를 프로젝트 루트로 보고 작성한다.
- 기존 보고서 스타일이 있으면 최근 파일 1개를 참고하되, 위 형식과 30줄 제한을 우선한다.
- 작업 내용이 확인되지 않으면 사용자에게 오늘 한 일을 짧게 물어본다.
