# 업무보고 작성

요청받은 날짜의 업무보고를 간결하게 작성한다. 날짜가 없으면 오늘 날짜를 사용한다. 특정 프로젝트명, 회사명, 절대경로를 가정하지 않는다.

## 절차

1. 현재 대화에서 완료한 작업을 파악한다.
2. 필요하면 `pwd`, `git status --short`, `git log --oneline -10`으로 최근 작업을 확인한다.
3. 기존 보고서 디렉토리를 찾고, 없으면 `docs/reports/업무보고/`를 만든다.
4. `업무보고_YYYYMMDD.md`에 보고서를 작성한다.
5. 생성 또는 수정한 파일과 핵심 내용을 짧게 요약한다.

## 저장 위치

아래 순서로 가장 먼저 맞는 위치를 사용한다.

1. 기존 `docs/reports/업무보고/`
2. 기존 `docs/reports/daily/`
3. 기존 `docs/daily/`
4. 기존 `reports/` 아래 `업무보고/`
5. 아무것도 없으면 `docs/reports/업무보고/`

## 형식

```markdown
# 업무보고 - YYYY-MM-DD

## 업무내용
1) 큰 분류 1
2) 큰 분류 2

## 진행상황

### 1.1) 작업명
- 핵심 한 줄
- 결과/수치

### 2.1) 작업명
- 핵심 한 줄

## To do list
1) 다음 작업
2) 다음 작업
```

## 작성 원칙

- 가능하면 전체 보고서는 본문 30줄 이내로 유지한다.
- 작업 항목당 bullet은 최대 2개로 제한한다.
- bullet 하나는 한 줄로 작성한다.
- `완료`, `정상 동작`, `반영`, `정리`처럼 결과 우선 표현을 쓴다.
- 앞으로 할 일은 `To do list`에만 적는다.
- 결과 자체가 아니라면 긴 코드 경로, 구현 세부사항, 배경 설명은 피한다.
- `업무내용` 번호와 `진행상황` 제목을 맞춘다. 예: `1)` -> `1.1)`, `1.2)`.
- 구현자가 아닌 관리자도 이해할 수 있는 용어를 사용한다.
- 기존 보고서 스타일이 있으면 최근 파일 1개를 참고하되, 간결한 형식을 우선한다.
