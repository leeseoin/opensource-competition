# 프로젝트 문서 생성/업데이트

특정 프로젝트명이나 도메인에 고정하지 않는다. 현재 저장소의 실제 코드, 문서, 사용자 제공 컨텍스트를 기준으로 프로젝트 문서를 점진적으로 만든다.

## 핵심 원칙

- 기존의 유용한 기록을 삭제하지 않는다.
- 프로젝트명이 문서에 필요할 때만 현재 저장소나 사용자 제공 내용에서 확인한 이름을 사용한다.
- 구현된 내용과 계획된 내용을 구분한다. 아직 구현 전이면 `planned` 또는 `예정`으로 표시한다.
- 검증하지 않은 API, 배포, 테스트, 인프라 주장을 사실처럼 쓰지 않는다.
- 날짜는 `YYYY-MM-DD` 형식을 사용한다.
- 문서 변경 후에는 변경한 파일과 변경 이유를 요약한다.

## 대상 문서

프로젝트마다 문서 구조가 다를 수 있으므로, 먼저 기존 구조를 확인하고 맞춘다.

일반 후보:

- `AGENTS.md`: Codex 및 기타 코딩 에이전트를 위한 저장소 작업 지침.
- `README.md`: 프로젝트 개요, 실행 방법, 운영 구조.
- `docs/PROJECT.md`: 아키텍처, 범위, API, 제품 결정사항.
- `docs/TODO.md`: 다음 작업, 보류 사항, 우선순위.
- `docs/WBS.md` 또는 `docs/WBS.csv`: 일정 또는 작업 분해.
- `docs/architecture/`: 아키텍처, 요청 흐름, 운영 구조, 설계 문서.
- `docs/reports/`: 날짜별 업무보고, 분석 문서, 회의록.
- `docs/reference/`: 외부 참고 자료나 요구사항 원문 정리.

저장소에 위 파일이 없으면 사용자 요청에 맞는 최소 문서만 새로 만든다.

## AGENTS.md 작성 흐름

사용자가 프로젝트 개요를 주거나 "AGENTS.md 템플릿", "프로젝트 지침", "에이전트용 문서"를 요청하면 아래 순서로 처리한다.

1. 저장소 구조와 기존 문서를 먼저 확인한다.
2. 기존 `AGENTS.md`가 있으면 유지할 내용과 오래된 내용을 구분한다.
3. 없으면 기본 템플릿을 만든다.
4. 사용자 제공 프로젝트 개요, 현재 코드 구조, docs 내용을 반영해 프로젝트별 지침으로 구체화한다.
5. 모르는 값은 추정하지 않고 `확인 필요` 또는 `planned`로 남긴다.

### AGENTS.md 기본 템플릿

필요한 섹션만 사용하고, 프로젝트에 맞지 않는 섹션은 생략한다.

```md
# AGENTS.md

## 프로젝트 개요

- 프로젝트명:
- 목적:
- 핵심 기술:
- 운영/배포 구조:

## 현재 상태

- 완료:
- 진행 중:
- 다음 작업:

## 실행/검증

- 로컬 실행:
- 테스트:
- 빌드:
- 배포:

## 프로젝트 구조

주요 디렉토리와 책임을 정리한다.

## 작업 원칙

- 기존 사용자 변경을 되돌리지 않는다.
- 코드와 문서를 실제 구현 상태에 맞춘다.
- 민감 정보는 커밋하지 않는다.
- 구현 전 항목은 planned로 표시한다.

## 문서 관리

- 현재 구조 이해:
- 아키텍처:
- 운영/배포:
- 업무보고:
- 다음 할 일:

## 주의 사항

- 확인 필요:
- 보류 정책:
```

## 문서 갱신 절차

1. 관련 파일을 먼저 읽는다.
   - 문서 작업이면 기존 문서와 연결된 소스/설정 파일을 같이 확인한다.
   - 아키텍처 문서면 실행 흐름, 설정, 배포 파일을 확인한다.
   - 업무보고면 최근 변경사항, 커밋 상태, 사용자 제공 작업 로그를 확인한다.
2. 현재 작업과 관련된 섹션만 수정한다.
3. 사용자 제공 컨텍스트가 길면 먼저 구조화한다.
   - 프로젝트 개요
   - 운영 구조
   - 현재 진행 상황
   - 프로젝트 구조
   - 주요 설정값
   - 문서 참조 우선순위
   - 다음 작업
4. 문서의 주장마다 근거가 필요한 경우 출처를 남긴다.
   - 예: `README.md`, `docker-compose.yml`, `src/main/...`, 사용자 제공 회의록.
5. 새 문서를 만들 때는 저장소의 기존 디렉토리 관례를 우선한다.
6. 최종 응답에서 변경 파일과 변경 이유를 간단히 설명한다.

## 스타일

- 한국어 문서는 간결하고 결과 중심으로 작성한다.
- 기술 문서는 사실 기반으로 구체적으로 작성한다.
- 목록은 너무 깊게 중첩하지 않는다.
- 문서 전체를 갈아엎기보다 필요한 부분만 작게 수정한다.
- 단, 기존 문서가 다른 프로젝트명/도메인에 묶여 있으면 재사용 가능한 표현으로 정리한다.

## 판단 기준

- `AGENTS.md`는 "이 저장소에서 다음 에이전트가 바로 일할 수 있게 하는 운영 지침"이다.
- `README.md`는 "사람이 프로젝트를 이해하고 실행할 수 있게 하는 입구"다.
- `docs/architecture/*`는 "왜 이런 구조인지와 실제 요청/데이터 흐름을 설명하는 설계 기록"이다.
- `docs/TODO.md`는 "아직 하지 않은 일과 의사결정 대기 항목"이다.
- `docs/reports/*`는 "특정 날짜/이슈의 작업 결과 보고"다.
