# 프로젝트 후보 비교

작성일: 2026-06-21  
상태: planned

## 후보 1: AgentPay Guard

### 한 줄 정의

AgentPay Guard는 AI Agent의 결제 요청을 사용자 의도, 예산, 허용 범위, 정책 기준으로 검증하고, 승인·차단·결제 결과를 블록체인 해시로 감사 가능하게 기록하는 오픈소스 보안형 결제 Agent PoC이다.

### 문제 상황

AI Agent가 사람을 대신해 상품을 찾거나 API, 데이터, 연산 자원, 디지털 서비스를 사용하는 흐름이 등장하고 있다. 이때 Agent가 결제 권한을 갖게 되면 다음 문제가 발생한다.

- 사용자가 허용하지 않은 결제를 실행할 수 있다.
- 예산 한도를 초과할 수 있다.
- prompt injection으로 악성 결제 요청이 유도될 수 있다.
- 어떤 의도와 조건으로 결제가 실행되었는지 나중에 확인하기 어렵다.
- 결제 요청, 정책 판단, 승인, 실행 결과의 감사 로그가 필요하다.

### 기존 흐름과의 차이

기존 결제는 사람이 직접 상품이나 서비스를 선택하고 결제한다.

```text
사람
  -> 상품/서비스 선택
  -> 결제 승인
  -> 결제 완료
```

Agent 결제에서는 사람이 큰 조건만 주고, Agent가 세부 선택과 결제 요청을 수행할 수 있다.

```text
사용자
  -> 예산/조건 위임
  -> AI Agent가 결제 후보 생성
  -> 정책 엔진이 검증
  -> 허용된 결제만 실행
  -> 결제 이력 감사
```

### 블록체인 사용 이유

블록체인은 결제를 직접 안전하게 만드는 장치가 아니라, 결제 의도와 실행 이력의 무결성을 증명하는 감사 레이어로 사용한다.

온체인 기록 대상:

- 사용자 의도 hash
- 결제 요청 hash
- 정책 판단 hash
- 승인 또는 차단 결과 hash
- 결제 결과 hash

### PoC 구현 범위

- 사용자 결제 의도 등록 API
- Agent 결제 요청 생성
- 예산/도메인/서비스/금액 정책 검사
- 결제 simulator 또는 x402 mock 연동
- 결제 승인/차단 기록
- AuditAnchor 스마트컨트랙트에 eventHash 기록
- 대시보드에서 결제 요청, 정책 판단, txHash 확인

### 시연 시나리오

```text
사용자: "날씨 데이터 API는 최대 1달러까지만 사용해."
Agent: API 호출 및 결제 요청 생성
Policy Engine: 허용 도메인과 금액 한도 확인
Payment Simulator: mock 결제 수행
Audit Anchor: 사용자 의도/정책 판단/결제 결과 hash 기록
Dashboard: 결제 성공, 차단 사유, txHash 확인
```

### 장점

- AI + 결제 + 블록체인 조합이 명확하다.
- x402, AP2 등 실제 시장 흐름과 연결하기 좋다.
- 시연이 비교적 직관적이다.
- 블록체인 역할이 감사와 무결성 증명으로 설명된다.

### 리스크

- 실제 결제와 연결하면 법적·보안적 리스크가 크다.
- x402나 AP2를 깊게 구현하려 하면 일정이 커질 수 있다.
- PoC에서는 mock 결제와 테스트넷으로 범위를 강하게 제한해야 한다.

### 통과 후 고도화

1. x402 실제 sandbox 또는 테스트넷 연동
2. 사용자 의도 서명 구조 도입
3. prompt injection 기반 결제 공격 방어
4. 예산·도메인·서비스별 정책 고도화
5. 결제 승인 워크플로우 추가
6. Agent 지갑/권한 관리
7. AP2 같은 결제 의도 증명 구조 조사 및 일부 반영

## 후보 2: OpenOps Guard

### 한 줄 정의

OpenOps Guard는 온프레미스 AI가 시스템 작업을 제안할 때 정책 기반으로 위험도를 판단하고, 승인된 제한 명령만 실행하며, 요청·승인·실행·결과 이력을 블록체인 해시로 검증하는 오픈소스 AI Ops 보안 플랫폼이다.

### 문제 상황

온프레미스 AI에게 시스템 제어 권한을 주면 다음 문제가 발생한다.

- AI가 위험한 명령을 생성할 수 있다.
- 프롬프트나 Markdown 규칙만으로는 정책 강제가 어렵다.
- 누가 어떤 작업을 승인했는지 추적해야 한다.
- 실행 로그가 나중에 조작되지 않았는지 검증해야 한다.
- 폐쇄망이나 내부망에서는 외부 SaaS에 운영 로그를 맡기기 어렵다.

### 기존 흐름과의 차이

기존 운영 자동화는 사람이 미리 정의한 job이나 playbook을 실행한다.

```text
운영자
  -> job/playbook 선택
  -> 실행
  -> 내부 로그 저장
```

OpenOps Guard는 AI가 작업 계획을 제안하되, 실제 실행은 정책 엔진과 executor가 통제한다.

```text
사용자 자연어 요청
  -> AI가 실행 계획 생성
  -> 정책 엔진 위험도 판단
  -> 필요 시 관리자 승인
  -> 제한된 명령만 실행
  -> 실행 이력 hash를 블록체인에 기록
```

### 블록체인 사용 이유

블록체인은 시스템을 직접 보호하는 방어 수단이 아니라, AI 제어 행위의 감사 로그가 조작되지 않았음을 증명하는 레이어로 사용한다.

온체인 기록 대상:

- 사용자 요청 hash
- AI 계획 hash
- 정책 판단 hash
- 승인 결과 hash
- 실행 명령 hash
- 실행 결과 hash
- 정책 버전 hash

### PoC 구현 범위

- 자연어 요청 등록 API
- AI 계획 생성 모듈
- allowlist/denylist 기반 정책 엔진
- 승인 API
- 제한된 command executor
- Audit DB
- AuditAnchor 스마트컨트랙트에 eventHash 기록
- 대시보드에서 실행 이력과 hash 검증 상태 확인

### 시연 시나리오

```text
사용자: "현재 서버 상태 확인해줘."
AI: CPU, 메모리, 디스크 조회 계획 생성
Policy Engine: READ_ONLY로 판단
Executor: 허용된 명령 실행
Audit Anchor: 요청/계획/실행 결과 hash 기록
Dashboard: 실행 결과와 txHash 확인
```

위험 명령 차단 시나리오:

```text
사용자: "/var/log 아래 파일 다 지워줘."
AI: 삭제 명령 후보 생성
Policy Engine: DENY로 판단
Executor: 실행하지 않음
Dashboard: 차단 사유와 감사 hash 표시
```

### 장점

- 보안 문제와 블록체인 감사 로그의 연결이 자연스럽다.
- 온프레미스 AI라는 목표와 잘 맞는다.
- 실제 운영 환경의 pain point가 명확하다.
- PoC 범위를 안전하게 제한하기 쉽다.

### 리스크

- 결제 Agent보다는 시장 트렌드 설명이 덜 화려할 수 있다.
- executor를 잘못 설계하면 보안 리스크가 생긴다.
- 실제 시스템 제어보다는 제한된 데모 환경으로 시작해야 한다.

### 통과 후 고도화

1. Ollama, vLLM, llama.cpp 기반 온프레미스 LLM 전환
2. OPA/Rego 또는 YAML 기반 정책 엔진 고도화
3. Docker/VM 기반 실행 샌드박스 강화
4. 다중 승인, Slack/Email 승인 연동
5. Merkle root 기반 대량 감사 로그 anchoring
6. Docker, systemd, Kubernetes 대상 확장
7. prompt injection, command injection, policy bypass 위협 모델 작성

| 항목 | AgentPay Guard | OpenOps Guard |

| 핵심 주제 | AI Agent 결제 보안 | 온프레미스 AI 시스템 제어 보안 |
| AI 역할 | 결제 요청 생성 | 시스템 작업 계획 생성 |
| 블록체인 역할 | 결제 의도와 결과 감사 | 제어 요청과 실행 결과 감사 |
| PoC 난이도 | 중간 | 중간 |
| 법적 리스크 | 상대적으로 높음 | 상대적으로 낮음 |
| 보안 리스크 | 결제 오남용 | 명령 실행 오남용 |

| 대회 차별성 | 최신 결제 트렌드와 연결 | 온프레미스 AI 보안과 연결 |

## 현실적인 선택 기준

### 결제 Agent를 하고 싶다면

AgentPay Guard가 더 맞다.

다만 실제 결제 기능을 만들기보다는, 다음 표현으로 제한해야 한다.

```text
AI Agent 결제 요청의 정책 검증과 감사 로그 PoC
```

실제 돈을 움직이는 결제 시스템이 아니라, 결제 권한을 가진 Agent를 안전하게 통제하는 보안 레이어로 잡아야 한다.

### 온프레미스 AI 제어를 하고 싶다면

OpenOps Guard가 더 맞다.

다만 실제 운영 서버에 위험 명령을 실행하는 시스템이 아니라, 제한된 명령 템플릿과 승인 흐름을 가진 AI Ops 제어 PoC로 잡아야 한다.

## 추천

개인적 추천은 다음과 같다.

1. 결제 Agent에 대한 동기와 흥미가 크다면 **AgentPay Guard**
2. 온프레미스 AI와 시스템 제어라는 장기 목표가 더 중요하다면 **OpenOps Guard**

대회 제출 관점에서 가장 안전한 선택은 OpenOps Guard이다. 반면 최신 트렌드와 결제 Agent라는 주제성은 AgentPay Guard가 더 강하다.

결제 Agent를 포기하기 어렵다면 AgentPay Guard를 선택하되, 범위를 반드시 "결제 실행"이 아니라 "결제 요청 검증, 한도 통제, 감사 로그"로 제한하는 것이 현실적이다.
